PROJECT OVERVIEW
----------------
User is porting The Legend of Zelda (NES) to Sega Genesis/Mega Drive.
Methodology follows mairtrus's SMB1 port: manual 68000 rewrite of all 6502 code,
verbatim reuse of NES data (level data, graphics tables, etc.).

All source files live in /home/claude/zelda_port/ in the container.
Key files:
  main.asm                    <- Genesis startup, vectors, JOYPAD_INIT
  vdp_layer.asm               <- PPU->VDP translation layer
  bank_FF_gen68k.asm          <- SOURCE for fixed bank (edit THIS, not _vdp.asm)
  bank_FF_gen68k_vdp.asm      <- AUTO-GENERATED by patch_vdp.py -- do not edit
  bank_00..06_gen68k.asm      <- SOURCE for translated NES switchable banks
  ram_map.asm                 <- RAM label declarations
  build.bat                   <- Windows build script (runs on user's machine)
  patch_vdp.py                <- Regenerates _vdp.asm from _gen68k.asm
  fix_syntax.py               <- Cleans up ca65 translator artifacts
  fix_checksum.py             <- Computes and writes Genesis ROM checksum

BUILD PIPELINE (user runs build.bat on Windows):
  1. patch_vdp.py    -> regenerates all bank_*_gen68k_vdp.asm
  2. fix_syntax.py   -> cleans up translator artifacts
  3. vasmm68k_mot.exe -> assembles main.asm -> zelda_raw.md
  4. fix_checksum.py -> writes correct checksum -> zelda.md

IMPORTANT: Always fix SOURCE files (main.asm, vdp_layer.asm, bank_*_gen68k.asm).
           The _vdp.asm files are regenerated each build and will overwrite edits.
           Binary patching zelda.md directly is a workaround only when vasm is
           unavailable (Linux container has no vasm, no network to download it).

REMAINING KNOWN BUGS
================================================================================

BUG A - MODERATE: VSRAM write command wrong (Y scroll)
  File: vdp_layer.asm, PPU_WRITE_2005 (second write = Y scroll)
  Current:  move.l #$40000010,($C00004)  -- writes to VRAM $0010, not VSRAM
  Correct:  move.l #$40004000,($C00004)  -- VSRAM write addr 0
  VSRAM write CD bits = 000101: CD2=1 sets bit14 -> $40004000
  Effect of bug: Y scroll data corrupts tile graphics. Won't crash.

BUG B - MODERATE: NES NMI handler not called
  File: vdp_layer.asm, VDP_VBLANK_HANDLER
  The translated NMI handler is commented out:
    ; jsr NMI_HANDLER   <- never called!
  NMI drives all game logic on NES. Game will appear frozen once it boots.
  Fix: bsr vec_0x01E494_NMI  (label in bank_FF_gen68k_vdp.asm)

BUG C - MODERATE: Sprite link field always 0
  File: vdp_layer.asm, VDP_OAM_DMA_TRANSFER
  Genesis sprites chain via link field. All links=0 means only sprite 0 shows.
  Fix: set link = sprite_index+1 for sprites 0..62, link=0 for sprite 63.

BUG D - MODERATE: Sprite tile index not offset to Genesis VRAM base
  NES tile 0 != Genesis tile 0 in VRAM after CHR data load.
  Sprite tile indices need an offset once CHR data is loaded at a VRAM address.

BUG E - FUTURE: MMC1 bank switching not implemented
  Zelda uses MMC1 mapper. MOVE.B D0, ROM_$8000 etc. in translated code does nothing.
  All 7 banks (bank_00..06) exist as 68000 code but aren't bank-switched yet.


================================================================================
VDP ADDRESS COMMAND REFERENCE
================================================================================

Formula for VRAM WRITE command (32-bit longword to $C00004):
  cmd = ((addr & 0x3FFF) << 16) | 0x40000000 | (addr >> 14)

Formula for CRAM WRITE:  0xC0000000 | (byte_offset << 16)
Formula for VSRAM WRITE: 0x40004000 | (byte_offset << 16)

QUICK SANITY CHECK on any VDP command:
  High nibble $4x-$7x = VRAM write (OK)
  High nibble $C0-$CF = CRAM write (OK)
  High nibble $40 with bit14 set ($4x where x has bit6) = VSRAM write (OK)
  High nibble $8x-$Bx = BIT7 SET IN HIGH WORD = READ command, not write (BAD)
  $BC000003 was BAD: $BC has bit7 set -> READ -> caused bus error

VERIFIED CORRECT COMMANDS:
  VRAM $0000 (tile base / VRAM clear):   $40000000  OK in code
  VRAM $BC00 (HScroll table):            $7C000002  FIXED session 3
  VRAM $C000 (Plane A nametable):        $40000003
  VRAM $D800 (sprite table):             $58000003  FIXED session 3
  VRAM $E000 (Plane B nametable):        $60000003
  CRAM $0000 (palette):                  $C0000000  OK in code
  VSRAM $0000 (vertical scroll):         $40004000  NOT YET FIXED


================================================================================
GENESIS HARDWARE MEMORY MAP
================================================================================
$000000-$0001FF  Exception vectors (64 x 4 bytes)
$000200+         ROM code
$A10001          Console version register (bit0=1 means TMSS present)
$A10003          Joypad port 1 data
$A10009          Joypad port 1 control (write $40 = strobe output enable)
$A11100          Z80 bus request ($0100=request bus, $0000=release)
$A11200          Z80 reset ($0100=assert reset, $0000=release)
$A14000          TMSS register (write 'SEGA' if version != 0)
$C00000          VDP data port
$C00004          VDP control port (write command/address; read status)
$C00008          H/V counter
$FF0000-$FFFFFF  68K RAM (64KB)

VDP STATUS WORD (read from $C00004):
  bit 3 = VBLANK active    <- NES $2002 bit7 maps here
  bit 5 = sprite collision <- NES $2002 bit6 maps here
  bit 6 = sprite overflow

VDP SHADOW REGS IN RAM ($FFEF00-$FFEF0A) -- NOT cleared by RAM clear:
  $FFFFEF00  PPUCTRL_SHADOW     NES $2000 shadow
  $FFFFEF01  PPUMASK_SHADOW     NES $2001 shadow
  $FFFFEF02  PPUADDR_HI         VRAM addr latch hi byte
  $FFFFEF03  PPUADDR_LO         VRAM addr latch lo byte
  $FFFFEF04  PPUADDR_LATCH      0=expect hi, 1=expect lo
  $FFFFEF05  PPUSCROLL_X        H scroll shadow
  $FFFFEF06  PPUSCROLL_Y        V scroll shadow
  $FFFFEF07  PPUSCROLL_LATCH    0=expect X, 1=expect Y
  $FFFFEF08  VRAM_ADDR_CURR     current VRAM address (word)
  $FFFFEF09  OAM addr shadow
  $FFFFEF0A  VDP_REG1_SHADOW    VDP mode reg 2 (display+vblank bits)

NES RAM in 68K space:
  $FF0000-$FF07FF  NES work RAM ($0000-$07FF)
  $FF0200-$FF02FF  NES OAM (64 sprites x 4 bytes)
  $FF00F0          Joypad 1 state (written by READ_JOYPAD)


================================================================================
CRASH LOG INTERPRETATION GUIDE
================================================================================

SR bits[10:8] = interrupt priority mask (IPM):
  SR=$2700 = IPM=7 = main code, no interrupts active
  SR=$2600 = IPM=6 = INSIDE level-6 VBlank handler
  SR=$2400 = IPM=4 = INSIDE level-4 HBlank handler

If PC=$2D4 always = stuck in EXCEPTION_HANDLER (BRA to self)
To find WHERE it crashed: check A7 (stack pointer) value in the log.
In BizHawk RAM watch, read longword at address in A7 -- that's the saved PC
(the address that was executing when the exception fired).

D0=D7=$FFFF = DBRA counters ran to -1 = those loops completed.
  D7=$FFFF from VDP_CLEAR_VRAM loop (MOVE.W #16383,D7 then DBRA)
  D0=$FFFF from RAM clear loop (MOVE.W #$3BFF,D0 then DBRA)
