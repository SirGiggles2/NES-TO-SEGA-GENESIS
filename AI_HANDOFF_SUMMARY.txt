OPERATIONAL GUIDE FOR THE NEXT AI
======================================================================

Read this first
- This repo is not a normal “rewrite Zelda in a higher-level engine” project.
- It is a low-level NES-to-Genesis port that preserves original Zelda logic and
  data layout as much as possible, then emulates the NES PPU contract on top of
  the Genesis VDP.
- The debugging style that works here is:
  1. make a very small source/data change
  2. build a new numbered ROM
  3. run batch diag and/or oracle trace
  4. compare artifacts, not gut feelings
- Do not assume a visual bug is in the renderer until the traces say so.
- Do not assume a sequence/music split is “just audio” until the traces say so.
- The project has repeatedly produced bugs where one register-clobber or one
  table-shape mistake manifests as “graphics”, “audio”, or “title logic.”

What the important directories are
- `src\main.asm`
  - Genesis entry point, vectors, crash/trace plumbing, and the inclusion point
    for the translated bank code and bridge.
- `src\bridge\vdp_layer.asm`
  - the heart of the port
  - emulates NES PPU register behavior using Genesis VDP
  - includes `PPU_WRITE_2000`, `PPU_WRITE_2006`, `PPU_WRITE_2007`,
    nametable rendering, palette conversion, CHR shadow flushes, and trace marks
- `src\banks\generated\`
  - “editable truth” for translated bank code
  - if you need to patch translated Zelda logic, patch here first
- `src\banks\generated_vdp\`
  - regenerated build products
  - these are recreated by `build\build.bat`
  - do not treat these as durable hand-edited source unless you intentionally
    understand that the next build regeneration may overwrite them
- `tools\`
  - all build helpers, batch builders, diff tools, and report summarizers live here
- `diag\scripts\`
  - BizHawk Lua scripts used for diag/oracle/focused traces
- `diag\reports\`
  - canonical output folder for all reports
- `diag\reports\Files I Want\`
  - convenience copy target for the user to inspect/share artifacts quickly
- `build\`
  - numbered output ROMs (`zelda_vNNN.md`) and corresponding listings (`.lst`)

How the translated-bank build flow actually works
- `build\build.bat` does all of this:
  1. runs `tools\patch_vdp.py`
     - regenerates `src\banks\generated_vdp\*.asm` from
       `src\banks\generated\*.asm`
  2. runs `tools\fix_syntax.py`
     - normalizes translated syntax for vasm
  3. assembles `src\main.asm` with `vasmm68k_mot.exe`
  4. fixes the Genesis checksum with `tools\fix_checksum.py`
  5. writes the next unused numbered ROM to `build\zelda_vNNN.md`
- Consequence:
  - if you only patch `generated_vdp`, your changes may be blown away on the next build
  - durable changes should usually go in `generated`

How to build a new ROM
- Command:
  - `.\build\build.bat`
- Output:
  - next unused ROM path: `build\zelda_vNNN.md`
  - matching listing: `build\zelda_vNNN.lst`
- Important:
  - version numbers are auto-assigned by scanning existing `build\zelda_v*.md`
  - do not guess the next number; let the script allocate it

How to run batch diagnostics
- Runner:
  - `run_batch.ps1`
- Example:
  - `.\run_batch.ps1 -StartRom zelda_v352`
- What it does:
  - rewrites `START_ROM_HINT` in `diag\scripts\zelda_diag_v9.lua`
  - launches BizHawk on the requested ROM
  - BizHawk loads the diag script and runs batch mode
  - the Lua script runs the requested ROM and the next `BATCH_SIZE-1` ROMs
  - current `BATCH_SIZE` is `5`
  - when `diag_batch_summary_v9.txt` updates, the PowerShell runner closes BizHawk
- Outputs:
  - `diag\reports\diag_report_vNNN.txt`
  - `diag\reports\diag_menu_vNNN.csv`
  - screenshots written by the diag script
  - copied convenience artifacts in `diag\reports\Files I Want\`
- Important implementation details:
  - `run_batch.ps1` creates `diag\scripts\zelda_diag_v9.lock`
  - if that lock remains after a crash, remove it manually before rerunning
  - the runner restores the Lua script afterward
- What the batch summary file is:
  - `diag\reports\diag_batch_summary_v9.txt`
  - this is the completion signal the runner trusts

How to run a single oracle trace
- Runner:
  - `run_oracle_trace.ps1`
- Example:
  - `.\run_oracle_trace.ps1 -Rom zelda_v352`
- What it does:
  - rewrites `TRACE_HINT` in `diag\scripts\zelda_oracle_trace.lua`
  - launches BizHawk on a single ROM
  - captures frame-by-frame CSV state
  - closes BizHawk automatically when the summary file updates
  - copies trace artifacts to `Files I Want`
- Outputs:
  - `diag\reports\oracle_trace_zelda_vNNN.csv`
  - `diag\reports\oracle_menu_zelda_vNNN.csv`
  - `diag\reports\oracle_trace_zelda_vNNN_summary.txt`
  - `diag\reports\oracle_events_zelda_vNNN.csv`
  - `diag\reports\oracle_writes_zelda_vNNN.csv`
- Important implementation details:
  - lock file:
    - `diag\scripts\zelda_oracle_trace.lock`
  - completion signal:
    - `diag\reports\oracle_trace_zelda_vNNN_summary.txt`
  - the runner rewrites the Lua file in-place, so do not run two oracle jobs in parallel

How to run a focused PPU/VDP write trace
- Runner:
  - `run_ppu_write_trace.ps1`
- Example:
  - `.\run_ppu_write_trace.ps1 -Rom zelda_v352`
- Purpose:
  - use this only when normal diag/oracle traces are too coarse
  - this is for answering “what exact shadow writes / VDP writes / render-cell
    words happened?”
- Outputs:
  - `diag\reports\ppu_writes_zelda_vNNN.csv`
  - `diag\reports\render_cells_zelda_vNNN.csv`
  - `diag\reports\chr_dirty_zelda_vNNN.csv`
  - `diag\reports\chr_focus_zelda_vNNN.csv`
  - completion sentinel:
    - `diag\reports\ppu_writes_zelda_vNNN.done`
- What each file is for:
  - `ppu_writes_...csv`
    - low-level writes: `vdp_ctrl`, `vdp_data`, `ppu_buf`, and now `pal_shadow`
  - `render_cells_...csv`
    - every live `PPU_RENDER_NAMETABLE_CELL` write with the renderer’s expected
      tileword and palette bits
  - `chr_dirty_...csv`
    - per-frame CHR tile dirtiness summary
  - `chr_focus_...csv`
    - very focused logging for suspicious title tiles

How to summarize batch reports quickly
- Tool:
  - `tools\diag_matrix.py`
- Example:
  - `py tools\diag_matrix.py`
- Purpose:
  - gives one-line summaries of many diag reports
  - shows score, NMI count, PPU count, sequence count, stuck PC, crash PC, etc.
- This is the fastest way to scan a new experiment batch before drilling deeper.

How to diff oracle traces
- Tool:
  - `tools\trace_diff.py`
- Example:
  - `py tools\trace_diff.py diag\reports\oracle_trace_zelda_v351.csv diag\reports\oracle_trace_zelda_v352.csv`
- Purpose:
  - find the first useful divergence between two oracle traces
  - best for “what changed in the live path?” questions
- Use oracle event CSVs directly when the main trace is too coarse and the key
  difference is event order rather than exported RAM state.

When to use each trace mode
- Use batch diag when:
  - you have 3-5 ROM variants and need quick comparative health
  - you care about high-level metrics and screenshots
- Use oracle trace when:
  - you need exact frame-by-frame state comparisons
  - you have narrowed the question to 1-2 ROMs
- Use focused PPU write trace when:
  - oracle proves the lane is alive but the rendering is still wrong
  - you need exact palette shadow writes, render-cell palette bits, or CHR mutations

What the key recurring metrics usually mean
- High `NMI completion count`
  - the build is alive; not necessarily correct
- `PPU event count = 0`
  - title/intro requests probably never woke or were fully suppressed
- `PPU event count = 24`
  - older “recovered but still legacy/fallback” title lane
- High hundreds of PPU events
  - often means the title/file-select path is much more awake, but still may be wrong
- `CPU STUCK at PC=$000F34`, `$000F3C`, or `$000FC2`
  - often heuristic noise, because these correspond to idle-loop-style states once
    the engine is alive
  - treat them as symptoms only if other metrics regress too
- `screen_ready`
  - this waking is a strong signal the title lane or follow-on lane advanced
- `PPU_REQ_INDEX / PPU_REQ_PTR / PPU_REQ_RES`
  - these are often the best “what buffer is really being requested?” clues in oracle
